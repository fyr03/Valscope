Bug: DATE-to-LONGTEXT JOIN Replaces an Existing Match with a Different Row
     After ANALYZE, Violating S1/S2 Monotonicity

Description:
This issue can be described using two temporal states of the same tables.

In the baseline state S1, after creating the tables and inserting a small
amount of data, the query returns exactly one row, with `c1 = 9000000`:

------------------------------------
mysql> SELECT 'S1' AS stage, id FROM s1_ids;
+-------+---------+
| stage | id      |
+-------+---------+
| S1    | 9000000 |
+-------+---------+
1 row in set
------------------------------------

Then I transform S1 into S2 by inserting 64 additional rows into the left
table, followed by ANALYZE TABLE, and execute the same query again. In S2,
the query still returns exactly one row, but now the row is `c1 = 9000006`:

------------------------------------
mysql> SELECT 'S2' AS stage, id FROM s2_ids;
+-------+---------+
| stage | id      |
+-------+---------+
| S2    | 9000006 |
+-------+---------+
1 row in set
------------------------------------

Because S2 is obtained from S1 by append-only inserts into the left table,
every row already present in the JOIN result in S1 should still be present in
S2. That monotonicity property is violated here: the old match `9000000`
disappears and is replaced by `9000006`.

------------------------------------
mysql> SELECT 'missing_from_s2' AS check_name, s1.id
    -> FROM s1_ids AS s1
    -> LEFT JOIN s2_ids AS s2
    ->   ON s1.id = s2.id
    -> WHERE s2.id IS NULL;
+-----------------+---------+
| check_name      | id      |
+-----------------+---------+
| missing_from_s2 | 9000000 |
+-----------------+---------+
1 row in set
------------------------------------

Observed summary:

------------------------------------
S1 result set = {9000000}
S2 result set = {9000006}
Missing from S2 = {9000000}
------------------------------------

This case is similar to other DATE-versus-text wrong-result bugs, but it is
distinguishable from the earlier reports in two ways:

1. It does not collapse to zero rows.
2. Instead, the query keeps one row in both S1 and S2, but the identity of
   the matched right-hand row changes from `9000000` to `9000006`.

The critical JOIN predicate is:

------------------------------------
sub1.d = sub2.txt
------------------------------------

where `sub1.d` is a DATE column and `sub2.txt` is a LONGTEXT column. The row
that disappears in S1 uses an invalid date-like LONGTEXT value, while the row
that appears in S2 uses a valid datetime-like LONGTEXT value.

How to repeat:
SET SESSION sql_mode = '';
SET SESSION cte_max_recursion_depth = 1000;

DROP DATABASE IF EXISTS repro_min_shift;
CREATE DATABASE repro_min_shift;
USE repro_min_shift;

DROP TABLE IF EXISTS t_main;
DROP TABLE IF EXISTS t_ref;
DROP TABLE IF EXISTS s1_ids;
DROP TABLE IF EXISTS s2_ids;

CREATE TABLE t_main (
    id INT NOT NULL PRIMARY KEY,
    c2 VARCHAR(255) NOT NULL,
    c4 INT NULL,
    d  DATE NOT NULL,
    KEY idx_d (d)
);

CREATE TABLE t_ref (
    id  INT NOT NULL PRIMARY KEY,
    c3  DECIMAL(10,2) NOT NULL,
    txt LONGTEXT NULL,
    dt  DATETIME NULL
);

-- S1: one match, expected result id = 9000000
INSERT IGNORE INTO t_main (id, c2, c4, d) VALUES
(1, '',           1, 'not-a-date'),
(2, '1',          4, 'not-a-date'),
(3, 'not-a-date', -1, 'not-a-date'),
(4, 'x',          0, '2023-01-01');

INSERT INTO t_ref (id, c3, txt, dt) VALUES
(9000000, 0.00, '/39ddgi4:fw7qr4vxsaqg', '2021-11-09 16:55:52'),
(9000006, 0.00, '2023-01-01 00:00:00',   '2005-04-12 04:37:25');

ANALYZE TABLE t_main;
ANALYZE TABLE t_ref;

CREATE TABLE s1_ids (
    id INT PRIMARY KEY
);

INSERT INTO s1_ids
SELECT DISTINCT r.id
FROM t_main AS m
JOIN t_ref  AS r
  ON m.d = r.txt
WHERE r.dt IS NOT NULL
  AND r.c3 = m.c2
  AND COALESCE(m.c4, 0) >= 1;

SELECT 'S1' AS stage, id
FROM s1_ids
ORDER BY id;

-- S2: append-only expansion of S1
INSERT IGNORE INTO t_main (id, c2, c4, d)
WITH RECURSIVE seq(n) AS (
    SELECT 1
    UNION ALL
    SELECT n + 1 FROM seq WHERE n < 64
)
SELECT
    1000 + n,
    'not-a-date',
    1,
    '2023-01-01'
FROM seq;

ANALYZE TABLE t_main;

CREATE TABLE s2_ids (
    id INT PRIMARY KEY
);

INSERT INTO s2_ids
SELECT DISTINCT r.id
FROM t_main AS m
JOIN t_ref  AS r
  ON m.d = r.txt
WHERE r.dt IS NOT NULL
  AND r.c3 = m.c2
  AND COALESCE(m.c4, 0) >= 1;

SELECT 'S2' AS stage, id
FROM s2_ids
ORDER BY id;

SELECT 'missing_from_s2' AS check_name, s1.id
FROM s1_ids AS s1
LEFT JOIN s2_ids AS s2
  ON s1.id = s2.id
WHERE s2.id IS NULL
ORDER BY s1.id;

-- Optional diagnostics
EXPLAIN FORMAT=TREE
SELECT DISTINCT r.id
FROM t_main AS m
JOIN t_ref  AS r
  ON m.d = r.txt
WHERE r.dt IS NOT NULL
  AND r.c3 = m.c2
  AND COALESCE(m.c4, 0) >= 1;
